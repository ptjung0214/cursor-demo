# 릴리스 노트 작성

두 git 태그 또는 커밋 사이의 변경을 분석해 사용자·개발자 모두 읽기 쉬운 한국어 릴리스 노트를 작성한다.

## 시작 전 확인

범위가 명시되지 않았으면 사용자에게 다음을 확인한다.

- **기준(from)**: 이전 태그/커밋 (예: `v1.2.0`, `abc1234`)
- **대상(to)**: 현재 태그/커밋 (예: `v1.3.0`, `HEAD`, `main`)
- **대상 독자**: 최종 사용자용 / 개발자용 / 둘 다

## 변경 수집

아래 명령을 병렬로 실행해 컨텍스트를 수집한다.

```bash
git log <from>..<to> --oneline --no-merges
git log <from>..<to> --format="%h %s%n%b" --no-merges
git diff <from>..<to> --stat
```

태그가 없으면 `git tag --sort=-v:refname`으로 최근 태그를 확인한다.

## 분류 기준

커밋 메시지와 diff를 함께 보고 아래 카테고리로 묶는다.

| 카테고리 | 포함 내용 | Conventional Commits 접두사 |
|----------|-----------|----------------------------|
| 새 기능 | 사용자에게 보이는 기능 추가 | `feat:` |
| 버그 수정 | 오동작·에러 수정 | `fix:` |
| 개선 | 성능, UX, 리팩터링 등 동작은 유지하며 나아진 변경 | `perf:`, `refactor:`, `style:` |
| 호환성 주의 | breaking change, 마이그레이션 필요 | `feat!:`, `BREAKING CHANGE` |
| 기타 | 문서, CI, 의존성 등 | `docs:`, `chore:`, `ci:` |

- 내부 리팩터링만 있고 사용자 영향이 없으면 **기타**에 두거나 생략한다.
- 같은 주제의 커밋은 한 항목으로 합친다.
- 커밋 메시지가 불명확하면 diff로 실제 변경을 확인한 뒤 서술한다.

## 출력 형식

아래 템플릿을 따른다. 버전·날짜는 사용자가 제공하거나 태그·커밋 날짜에서 가져온다.

```markdown
# vX.Y.Z (YYYY-MM-DD)

## 요약
[이번 릴리스의 핵심을 1~2문장으로]

## 새 기능
- ...

## 버그 수정
- ...

## 개선
- ...

## 호환성 주의
- ...

## 기타
- ...
```

### 작성 규칙

- **한국어**로 작성한다. 기술 용어(API, JWT 등)는 원문을 유지해도 된다.
- 각 항목은 **사용자 관점**으로 쓴다. "validator.js 수정" 대신 "이메일 형식 검증 오류 메시지 개선".
- 항목이 없는 섹션은 **섹션 자체를 생략**한다.
- breaking change가 있으면 **호환성 주의**를 반드시 포함하고, 필요한 조치를 명시한다.

## 완료 후

1. 변경 범위(`from` → `to`)와 커밋 수를 한 줄로 명시한다.
2. 사용자가 요청했으면 `CHANGELOG.md`에 추가하거나 별도 파일로 저장한다.
3. 배포 준비 맥락이면 테스트·마이그레이션·롤백 시 유의점을 짧게 덧붙인다.
